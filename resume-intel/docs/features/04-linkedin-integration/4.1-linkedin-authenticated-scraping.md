## 1. Problem Statement

DuckDuckGo search returns LinkedIn profile URLs but not the profile content. To get the candidate's full employment history, education, and contact info from LinkedIn, the app needs to visit those profile pages. LinkedIn requires authentication to view most profile data, so the app uses the recruiter's own LinkedIn session to scrape profiles — appearing as normal user browsing activity.

## 2. Goals

- Visit LinkedIn profile URLs found in DDG search results using the recruiter's authenticated session
- Extract name, headline, location, experience, education, and contact info
- Do so invisibly (hidden browser window) without the recruiter needing to interact
- Handle LinkedIn's dynamic DOM gracefully

## 3. Non-Goals

- Creating or modifying any LinkedIn content
- Scraping search results from LinkedIn directly (only visiting known profile URLs)
- Bypassing LinkedIn's authentication — the recruiter must log in themselves
- Bulk profile scraping beyond what DDG returns (max 5 URLs per candidate)

## 4. Personas & User Stories

- **As a recruiter**, I want the app to pull the candidate's LinkedIn profile automatically so that I can confirm I found the right person before reaching out.
- **As a recruiter**, I want to see the candidate's current title and employer from LinkedIn so that I can verify it matches the resume.

## 5. Functional Requirements

- **FR-1.** The system MUST only scrape LinkedIn profile URLs of the form `linkedin.com/in/{slug}` — never search pages or other LinkedIn sections.
- **FR-2.** The system MUST use the session cookies stored by feature 4.2 (LinkedIn Login) to authenticate.
- **FR-3.** The system MUST open a hidden `BrowserWindow` (not Playwright) using Electron's built-in Chromium for this scraping to avoid Playwright detection issues.
- **FR-4.** The system MUST extract: `name`, `headline`, `location`, `about`, `experience` (array, max 5 entries), `education` (array), `contactInfo` (if publicly visible).
- **FR-5.** The system MUST add a randomized 5-10 second delay between profile visits.
- **FR-6.** The system MUST close the hidden BrowserWindow after each profile visit.
- **FR-7.** If no LinkedIn session exists, the system MUST skip this step and show "LinkedIn not connected" in the UI — not throw an error.
- **FR-8.** The system MUST handle a LinkedIn login redirect (session expired) by clearing stored cookies and prompting the user to reconnect.
- **FR-9.** The system SHOULD attempt at most 2 LinkedIn profile URLs per candidate (the top 2 from DDG results).

## 6. Non-Functional Requirements

- **Legal** — This feature uses the recruiter's own LinkedIn session. The recruiter accepts LinkedIn's ToS on their own behalf. The app must not misrepresent this.
- **Performance** — Each profile visit MUST complete within 15 seconds.
- **Stealth** — Use realistic delays between requests. Do not set a custom User-Agent string.
- **Reliability** — If LinkedIn scraping fails, the pipeline continues with whatever data was gathered.

## 7. Acceptance Criteria

- **AC-1.** *Given* a valid LinkedIn session and a profile URL, *when* scraping runs, *then* name, headline, and location are returned.
- **AC-2.** *Given* no LinkedIn session is stored, *when* scraping is attempted, *then* the step is skipped and `linkedin_data` is stored as null.
- **AC-3.** *Given* LinkedIn redirects to the login page (expired session), *when* this is detected, *then* stored cookies are cleared and the user is notified to reconnect.
- **AC-4.** *Given* a profile URL that returns a 404, *when* visited, *then* the result is skipped and the next URL is tried.

## 8. Data Model

LinkedIn data shape (stored in `linkedin_data` JSON column):

```typescript
{
  profileUrl: string,
  name: string | null,
  headline: string | null,
  location: string | null,
  about: string | null,
  experience: Array<{
    title: string,
    company: string,
    duration: string
  }>,
  education: Array<{
    school: string,
    degree: string
  }>,
  contactInfo: string | null,
  scrapedAt: string  // ISO timestamp
}
```

## 9. API Surface

**File:** `src/main/search/linkedin.js`

```typescript
scrapeLinkedInProfile(profileUrl: string): Promise<LinkedInData | null>
```

Called by `ipc-handlers.js` after DDG search returns LinkedIn URLs.

## 10. UI / UX

`SearchStatus.jsx` progress step: "Checking LinkedIn..."

In `CandidateDetail` drawer — "LinkedIn Profile" section:
- Profile data displayed in a card
- "View on LinkedIn" button opens URL in system browser
- If null: "LinkedIn profile not found or not connected"
- If session expired: amber banner "LinkedIn session expired — reconnect in Settings"

## 11. AI / ML Considerations

No AI in this feature. Raw HTML is scraped and structured using DOM selectors. The structured LinkedIn data is later passed to the AI summarizer (2.3).

## 12. Integration Points

- `src/main/search/duckduckgo.js` — provides profile URLs

- `src/main/index.js` — BrowserWindow creation

## 13. Dependencies & Sequencing

- Must ship after: 3.1 (DDG must find LinkedIn URLs first), 4.2 (login must happen before scraping)
- Must ship before: 2.3 (summarizer needs LinkedIn data)

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| LinkedIn changes DOM structure | H | H | Use multiple fallback selectors per field; log raw HTML on failure |
| LinkedIn detects and bans the recruiter's account | L | H | Use realistic delays; warn user in Settings about ToS risk |
| Session expires during batch processing | M | M | Detect login redirect, clear cookies, notify user, skip remaining LinkedIn steps |
| Wrong person identified from DDG results | M | M | Show candidate name from LinkedIn prominently so recruiter can verify before outreach |
