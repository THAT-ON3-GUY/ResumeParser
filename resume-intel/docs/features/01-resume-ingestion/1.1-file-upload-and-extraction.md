## 1. Problem Statement

Recruiters receive scrubbed resumes as PDF and DOCX files. The app must accept these files, extract their raw text content, and hand it off to the AI parsing pipeline. Without reliable ingestion, no downstream feature works. Currently single-file upload works but batch upload, progress feedback, and error handling for corrupt or scanned files are missing.

## 2. Goals

- Accept PDF and DOCX files via drag-and-drop or file picker
- Support batch upload of up to 50 files simultaneously
- Extract clean raw text from each file
- Surface clear errors when a file cannot be read
- Hand extracted text to the AI parser pipeline immediately

## 3. Non-Goals

- OCR for scanned/image-only PDFs (future feature)
- Support for .doc (legacy Word), .odt, .txt, or any other format
- Cloud storage or file syncing
- File preview or thumbnail rendering

## 4. Personas & User Stories

- **As a recruiter**, I want to drag a folder of resumes onto the app so that I can process them all at once without clicking each one individually.
- **As a recruiter**, I want to see a progress indicator per file so that I know the app is working and which files are done.
- **As a recruiter**, I want a clear error message when a file fails so that I know to handle it manually rather than assuming it was parsed.

## 5. Functional Requirements

- **FR-1.** The system MUST accept `.pdf` and `.docx` files via drag-and-drop onto the `UploadZone` component.
- **FR-2.** The system MUST accept `.pdf` and `.docx` files via a file picker dialog triggered by clicking the upload zone.
- **FR-3.** The system MUST support uploading up to 50 files in a single batch operation.
- **FR-4.** The system MUST extract raw text from PDF files using `pdf-parse` in the main process.
- **FR-5.** The system MUST extract raw text from DOCX files using `mammoth` in the main process.
- **FR-6.** The system MUST display a per-file status indicator: Queued → Extracting → Parsing → Done / Error.
- **FR-7.** The system MUST reject files that are not `.pdf` or `.docx` and display a clear rejection message.
- **FR-8.** The system SHOULD display a warning when extracted text is fewer than 100 characters, indicating a likely scanned or corrupt file.
- **FR-9.** The system MUST pass the extracted text and original file name to `aiProvider.extractResume()` immediately after extraction.
- **FR-10.** The system MAY allow the user to retry a failed file without re-uploading.

## 6. Non-Functional Requirements

- **Performance** — Each file extraction MUST complete within 3 seconds for files under 5MB.
- **Security** — File reading happens in the main process only. The renderer never has access to raw file paths or contents directly — only via IPC.
- **Privacy** — Files are never uploaded to any server. All processing is local.
- **Reliability** — A failure on one file in a batch MUST NOT stop processing of remaining files.
- **Accessibility** — Upload zone MUST be keyboard accessible and include ARIA labels for screen readers.

## 7. Acceptance Criteria

- **AC-1.** *Given* a valid PDF file, *when* the user drags it onto the upload zone, *then* the file status changes to "Extracting" and raw text is returned within 3 seconds.
- **AC-2.** *Given* a valid DOCX file, *when* the user clicks the upload zone and selects it, *then* the file status changes to "Extracting" and raw text is returned.
- **AC-3.** *Given* a batch of 10 PDF files, *when* the user drops them all at once, *then* all 10 are processed in parallel and each shows its own status indicator.
- **AC-4.** *Given* a `.txt` file, *when* the user drops it, *then* the app displays "Unsupported file type" and does not attempt to process it.
- **AC-5.** *Given* a scanned PDF with no text layer, *when* extraction completes, *then* the app shows a warning "File may be scanned — text extraction returned minimal content."
- **AC-6.** *Given* one corrupt file in a batch of five, *when* processing runs, *then* the four valid files complete successfully and the corrupt file shows an error state.

## 8. Data Model

No new database tables — extracted text is passed directly to the parser and then stored by feature 9.1. The in-memory shape passed via IPC:

```typescript
{
  fileName: string,         // original file name e.g. "resume_john.pdf"
  filePath: string,         // absolute local path
  rawText: string,          // extracted plain text
  fileType: "pdf" | "docx",
  extractedAt: string,      // ISO timestamp
  charCount: number         // length of rawText for warning threshold
}
```

## 9. API Surface

IPC channels used by this feature:

| Channel | Direction | Payload |
|---|---|---|
| `resume:parse` | renderer → main | `{ filePath: string, fileName: string }` |
| `resume:parse-batch` | renderer → main | `{ files: Array<{ filePath, fileName }> }` |

Main process handler lives in `src/main/ipc-handlers.js`. File reading logic lives in `src/main/parser/fileReader.js`.

## 10. UI / UX

**UploadZone component** (`src/renderer/src/components/UploadZone.jsx`):
- Large centered drop target when no files are loaded
- Dashed border, icon, and text: "Drop resumes here or click to browse"
- Accepts only `.pdf` and `.docx` (enforced via `accept` attribute and drop handler filter)
- On drag-over: border highlights blue
- On drop: immediately shows file list with status badges

**File list** (below upload zone or replacing it):
- One row per file: filename, file type badge, status badge, error message if applicable
- Status badges: gray (Queued), blue (Extracting), purple (Parsing AI), green (Done), red (Error)
- Retry button on error rows

**Empty state:** Large upload zone centered on screen with icon and instructions.

**Error state:** Inline per file — never a blocking modal.

## 11. AI / ML Considerations

This feature does not call AI directly. It prepares text for `aiProvider.js`. See feature 2.1 for AI details.

## 12. Integration Points

- `src/main/parser/fileReader.js` — owns PDF and DOCX extraction logic
- `src/main/ipc-handlers.js` — registers `resume:parse` and `resume:parse-batch` handlers
- `src/renderer/src/components/UploadZone.jsx` — UI entry point
- Feature 2.1 (AI Extraction) — receives output of this feature

## 13. Dependencies & Sequencing

- Must ship before: 2.1, 2.2, all search features

## 14. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Scanned PDFs return empty text | H | M | Warn user, suggest manual copy-paste fallback |
| pdf-parse fails on encrypted PDFs | M | M | Catch error, show "File may be password protected" |
| Batch of 50 files blocks the main process | M | H | Use async queue with concurrency limit of 5 |
| Playwright Chromium conflicts with Electron Chromium | L | H | File reading uses Node.js only — no Playwright involvement here |

## 15. Rollout Plan

- No feature flag needed — this is core functionality
- Ships as part of Milestone 1 (already partially complete)
- Rollback: revert `fileReader.js` and `ipc-handlers.js`

## 16. Test Plan

- **Unit** — `fileReader.js`: test PDF extraction, DOCX extraction, empty file, corrupt file, encrypted PDF
- **Integration** — IPC round trip: drop file in renderer → main processes → status update returns
- **End-to-end** — Drop a real PDF resume, confirm raw text appears in console/DB
- **Manual** — Test with 50 files at once; test with a scanned PDF; test with a .txt file rejection

## 17. Documentation & Training

- One-paragraph user guide: "How to upload resumes"
- In-app tooltip on the upload zone explaining supported formats

## 18. Open Questions

1. Should we support drag-and-drop of an entire folder, or only individual files?
2. Is there a maximum file size we should enforce? (Suggested: 20MB per file)
3. Should batch processing be sequential or parallel? (Parallel with concurrency cap of 5 recommended)
